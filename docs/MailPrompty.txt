KájovoMail — prompty pro Codex CLI agenty

Tento soubor obsahuje hotové prompty pro dva agenty:
- lokální Codex CLI agent v repozitáři `c:\github\kajovomail`
- Codex CLI agent přímo na Ubuntu serveru

Používej je v uvedeném pořadí. Každý prompt je určený k přímému vložení do Codex CLI. Prompty jsou psané tak, aby agent pracoval samostatně, neptal se na zbytečné potvrzení a po každém kroku provedl vlastní kontrolu.

Společná pravidla pro všechny prompty
1. Jediný zdroj pravdy pro produktové a architektonické požadavky je `docs/KajovoMailZadaniFinal.md`.
2. Návrhový draft ignoruj. Nepoužívej `docs/KajovoMailZadaniDraft.md`.
3. Manifest designu uložený v `docs/` a veškeré assety v `brand/` jsou závazné pro brand, signaci, logo, palety, tokeny a motion.
4. Cílový produkční hostname je `mail.hcasc.cz`.
5. Na Ubuntu serveru už běží `hotel.hcasc.cz`, `dagmar.hcasc.cz` a `api.hcasc.cz`. Nesmí být poškozeny, vypnuty ani přepsány.
6. Implementuj skutečný software. Neponechávej `TODO`, `TBD`, stuby, placeholder texty ani falešné obrazovky.
7. Dodrž server-centric architekturu, žádné přímé volání OpenAI z klientů a žádná perzistentní lokální databáze pošty na desktopu ani Androidu.
8. Po každém promptu proveď relevantní testy, oprav nalezené chyby, udělej smysluplný commit a pokud existuje vzdálený GitHub repozitář, změny pushni.
9. Když chybí externí credential nebo jiný tvrdý blokátor, dokonči vše, co dokončit jde, a na konci vypiš jediný stručný blokátor s přesným názvem chybějící věci.

2. Inicializuj git, pokud ještě není inicializovaný. Použij výchozí branch `main`.
3. Přidej repo hygiene soubory pro monorepo: `.gitignore`, `.editorconfig`, `.gitattributes`, případně další nezbytné soubory pro konzistentní textové formáty a ignorování build artefaktů.

PROMPT 03 — LOKÁLNÍ AGENT — mailový backend, synchronizace, AI a nabídky
============================================================
Navazuj na existující backend. Jediný zdroj pravdy je `docs/KajovoMailZadaniFinal.md`. Dodrž server-centric architekturu, žádná přímá OpenAI volání z klientů, žádná lokální perzistentní databáze pošty v klientech.

Dodej funkční backendovou doménu KájovoMailu.

Proveď toto:
1. Implementuj moduly pro e-mailové účty, capability discovery, test připojení, bezpečné uložení credentialů a podporu více účtů na uživatele.
2. Implementuj IMAP jako plnohodnotný režim a POP3 jako omezený režim s jasně odlišenými capability a omezeními v API i datech.
3. Implementuj synchronizační model: počáteční napojení účtu, inkrementální sync, ruční refresh, periodický background sync, retry, idempotenci, obnovu po přerušení a promítání změn přes event vrstvu.
4. Implementuj strom složek, default/system vs user folders, create/rename/delete/move, oblíbené složky a správu omezení podle capability provideru.
5. Implementuj list/detail zpráv, threading podle RFC hlaviček, mark read/unread, flag/unflag, move, copy, archive, delete, bulk operace a virtuální pohledy Nepřečtené, Označené vlajkou a S přílohami.
6. Implementuj serverové vyhledávání včetně syntaxe `AND`, `OR`, `NOT`, přesné fráze v uvozovkách, scope current folder, current account a all folders. Výsledky musí být stránkované a stabilně řazené. Je přípustná kombinace provider search a pomocného serverového indexu. Rozhodnutí zdokumentuj.
7. Implementuj drafty, autosave, compose, reply, reply all, forward a idempotentní odesílání. Každý odchozí e-mail musí vzniknout jako `multipart/alternative` v požadovaném pořadí `text/plain` a `text/html`.
8. Implementuj konzervativní HTML rendering a validaci tak, aby nevznikaly zakázané prvky, externí obrázky, inline logo ani rizikové HTML.
9. Implementuj přílohy jako streamované přenosy přes backend.
10. Implementuj AI orchestrace přes OpenAI Responses API se structured output nebo ekvivalentně přísným schématem. Backend musí generovat strukturovaný meziformát a z něj deterministicky vytvořit plain text, HTML a preview. Produkční default nastav na `store: false`.
11. Implementuj ochranu AI soukromí, schema validaci, request policy a audit/metadata logy bez úniku citlivého obsahu do běžných logů.
12. Implementuj funkci `Nabídky`, vazbu na thread nebo message, povinná pole, auditované přechody stavů a API endpointy.
13. Implementuj WebSocket endpoint `/api/v1/events/ws` nebo kompatibilní event stream splňující zadání.
14. Rozšiř OpenAPI tak, aby pokrývalo minimální endpointy ze zadání.
15. Přidej integrační testy konektorů. Pro automatizované testy použij reálný test harness, například GreenMail v kontejneru, nebo srovnatelně věrné testovací prostředí pro IMAP, POP3 a SMTP. Přidej testy sync/retry, MIME generování, AI schema validace, Offers workflow a kritických error stavů.
16. Aktualizuj dokumentaci a testovací matici podle reálně implementovaného stavu.
17. Udělej commit a push.
